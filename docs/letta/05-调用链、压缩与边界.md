# `letta`：调用链、压缩与边界

## 1. conversation 创建主链

conversation 创建或分叉时，`ConversationManager` 会重建 system message，把它作为 message 0 写入，再通过 `conversation_messages` 维护真实顺序。后续 `AgentManager.get_context_window()` 再基于该 conversation 做窗口计算。

### 源码锚点

- `letta/services/conversation_manager.py:222-320`
- `letta/services/conversation_manager.py:613-713`
- `letta/services/agent_manager.py:3561-3595`

## 2. 摘要压缩主链

`Summarizer` 提供两条路径：固定缓冲裁剪和部分驱逐摘要。它负责把 transcript 送给 LLM 生成 summary message，用于在长对话时减小上下文窗口。

### 源码锚点

- `letta/services/summarizer/summarizer.py:36-488`

## 3. archivial search 主链

archive / passage 的检索入口统一走 `search_agent_archival_memory_async()`，它会先处理时间和时区过滤，再选择 SQL 或 TPUF 的搜索路径。

### 源码锚点

- `letta/services/agent_manager.py:2534-2620`

## 4. 关键边界

- core blocks memory、archive/passage 和记忆仓库 git 路径是并行层。
- git-backed memory 不是 archive 的另一种名字。
- 消息顺序、system prompt 和摘要消息共同参与上下文窗口，而不是由单一 message 列表决定。

### 源码锚点

- `letta/schemas/memory.py:68-688`
- `letta/services/archive_manager.py:502-542`
- `letta/services/block_manager_git.py:30-485`
- `letta/services/context_window_calculator/context_window_calculator.py:167-249`
