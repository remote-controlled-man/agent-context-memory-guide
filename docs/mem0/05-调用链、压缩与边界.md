# `mem0`：调用链、压缩与边界

## 1. `add()` 的调用链

`add()` 的核心流程是：

1. 解析 filters 与 `session_scope`
2. 读取最近消息窗口与已有 memories
3. 组装抽取 prompt
4. 调用 LLM 抽取
5. 去重、embed、写向量库
6. 写 SQLite history/messages
7. 更新 entity links

### 源码锚点

- `mem0/memory/main.py:424-618`
- `mem0/memory/main.py:640-1400`

## 2. `search()` 的调用链

搜索链路是：

`向量语义检索 -> keyword_search -> BM25 归一化 -> entity boost -> rerank`

因此它不是纯向量搜索。

### 源码锚点

- `mem0/memory/main.py:959-1152`

## 3. 异步实现的边界

`AsyncMemory` 不是另一套算法，主要通过异步包装和 `to_thread()` 复用同步逻辑。因此在行为分析上，同步版和异步版可以视为同一实现的两种运行方式。

### 源码锚点

- `mem0/memory/main.py:1687-1687`

## 4. 关键边界

- 没有完整 conversation compaction 主线。
- `history/messages` 是辅助上下文，不是长期 memory 本体。
- procedural memory 是单独的摘要式写入分支。

### 源码锚点

- `mem0/memory/main.py:1432-1432`
- `mem0/memory/storage.py:11-257`
