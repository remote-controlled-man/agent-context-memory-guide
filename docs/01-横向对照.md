## 3. 按维度看差异

### 3.1 `context` 谁来组装

- `claude code`：更偏“会话级缓存 + 文件注入”，`memdir` 与 `claudemd` 进入 prompt。
- `codex`：更偏“turn 级差分注入”，首次全量、之后增量。
- `hermes-agent`：更偏“prompt builder + provider hook”。
- `langmem`：更偏“工作流节点”，上下文本身就是 graph state。
- `letta`：更偏“Memory.compile() -> PromptGenerator -> ContextWindowCalculator”。
- `mem0`：更偏“为 memory 抽取服务的 prompt 上下文”，不是完整对话 runtime。

### 3.2 `summary / compaction` 谁负责

- `claude code`：压缩同时影响用户上下文缓存、session memory、消息视图。
- `codex`：压缩直接重写历史和 window generation。
- `hermes-agent`：压缩是核心能力，带 tool output 瘦身和 summary fallback。
- `langmem`：摘要就是产品能力本身。
- `letta`：摘要器负责驱逐旧消息时生成 summary message。
- `mem0`：没有统一的对话压缩主线，重点在抽取式记忆生成。

### 3.3 `history` 和 `memory` 是否分离

- 明显分离：`claude code`、`codex`、`hermes-agent`
- 半分离但同平台管理：`letta`
- 以 store 为主、history 不是第一对象：`langmem`
- SQLite history 主要服务抽取，不等于 long-term memory：`mem0`

### 3.4 `long-term memory` 的落点

- 文件目录：`claude code`
- SQLite state + 文件同步：`codex`
- 本地文件 + provider backend：`hermes-agent`
- `BaseStore` namespace：`langmem`
- blocks/archive/git repo：`letta`
- 向量库 + entity store：`mem0`

### 3.5 多 agent / 多线程关系

- `codex` 与 `letta` 的线程/会话/子 agent 关系最结构化。
- `claude code` 与 `hermes-agent` 有多 agent 能力，但 memory 仍主要围绕 session/project。
- `langmem` 和 `mem0` 更像被别的 agent runtime 调用的能力层。

## 4. 阅读建议

- 如果你关心“终端 agent 怎样做 context 差分和压缩”，优先看 `codex` 与 `claude code`。
- 如果你关心“插件式外接 memory provider”，优先看 `hermes-agent`。
- 如果你关心“平台如何统一消息、block memory、archive 与 git-backed memory”，优先看 `letta`。
- 如果你关心“怎么把 memory 能力嵌到自己的 agent/workflow 里”，优先看 `langmem` 与 `mem0`。
