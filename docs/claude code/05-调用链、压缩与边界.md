# `claude code`：调用链、压缩与边界

## 1. `/compact` 的主调用链

`/compact` 会先尝试 `SessionMemory` compaction，再走 `microcompact` / 传统压缩路径。执行成功后，它会清 `getUserContext()` 缓存、标记 post-compaction，并压掉相应警告。

这意味着压缩不仅改变消息窗口，也会直接影响后续 context 的重建方式。

### 源码锚点

- `commands/compact/compact.ts:46-117`
- `services/compact/sessionMemoryCompact.ts:1-1`

## 2. compact boundary 的意义

`/context` 和 API 视图都依赖 compact boundary。被 boundary 之前的消息不会继续作为“当前 prompt 历史”参与可视上下文分析，因此 compact 本质上会重写“哪些消息还属于当前可见上下文”。

### 源码锚点

- `utils/messages.ts:4608-4643`
- `commands/context/context.tsx:18-18`

## 3. 关键边界

- `history.ts` 是命令历史，不是语义记忆。
- `memdir` 是跨会话 memory。
- `SessionMemory` 是单会话摘要。
- `/context` 是 API 视图，不是 UI 原始视图。

阅读源码时如果不先把这四条边界分开，最容易把不同层面的“记忆”误合并。
