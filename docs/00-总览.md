# `repos/` Agent Context / Memory 源码分析总览

## 1. 分析范围

本目录覆盖 `E:\Agents\agent-memory\repos` 下 6 个仓库：

- `claude code`
- `codex`
- `hermes-agent`

- `letta`

目标不是复述项目说明，而是基于源码、测试、配置、migration、运行时路径解析逻辑，分析这些项目的：

- `context` 组装方式
- `summary / compaction` 机制
- `history` 与 `session/thread` 持久化
- `short-term memory` 与 `long-term memory`
- `检索路径`
- `state` / 存储后端
- 多 agent / 多线程 / 多 session 关系

## 2. 阅读方法

### 2.1 术语约定

- `context`：某一轮真正送进模型、或即将送进模型的可见上下文。
- `history`：消息历史、turn 历史、命令历史、thread transcript 等历史记录容器。
- `summary / compaction`：为缩短上下文窗口而进行的摘要、裁剪、压缩、窗口重写。
- `short-term memory`：面向当前 session/thread 的近程记忆，例如运行中缓存、当前会话摘要、最近消息窗口。
- `long-term memory`：跨 session/thread 持续存在、可被重新召回的记忆。
- `state`：除消息正文外的持久化运行时状态，例如 SQLite 元数据、线程表、job 表、路径布局、memory root。

### 2.2 证据原则

- 结论只以源码和测试为准。
- 如果某个行为主要由测试揭示，文档中会明确标成“测试验证到的行为”。
- 每份专题文档的每一节末尾都附有 `源码锚点`，用于回到实现位置核对。

## 3. 仓库分类

### 3.1 终端型 / 交互式 Agent Runtime

- [claude code](./claude%20code.md)
- [codex](./codex.md)
- [hermes-agent](./hermes-agent.md)

这三类项目都不是单一 memory 库，而是完整 agent runtime。阅读时要优先区分：

- 模型可见 `context`
- `history` / transcript
- 运行时 `state`
- 可跨会话复用的 memory

### 3.2 Memory / Context 工具库

- [langmem](./langmem.md)
- [mem0](./mem0.md)

这两类项目更像“可嵌入的 memory 能力层”：

- `langmem` 偏 `LangGraph/BaseStore` 场景，关注摘要节点、memory 抽取器、工具封装、命名空间与后台反射。
- `mem0` 偏“抽取式长期记忆”，关注向量库、SQLite 历史、实体链接、混合检索与 procedural memory。

### 3.3 Agent 平台 / 服务器型系统

- [letta](./letta.md)

`letta` 同时包含：

- block/core memory
- 会话与消息 ORM
- archive/passage 长期记忆
- 可选 git-backed memory

因此它既有 runtime，又有平台化持久层。

## 7. 交叉阅读提示

- `claude code` 与 `codex` 都同时存在 `context`、`history`、`compaction` 和单独的 memory 管线，但设计边界不同。
- `hermes-agent` 与 `letta` 都有“多条并行 memory 路径”，前者偏插件 provider，后者偏平台内建模型。
- `langmem` 与 `mem0` 都是 memory 能力层，但 `langmem` 偏工作流节点与 store namespace，`mem0` 偏抽取、去重、检索与存储。
